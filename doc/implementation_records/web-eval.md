# `/eval` 跨卡池单方案评估页面计划

## Summary

新增 `/gacha` 和 `/eval` 两个 Web 页面入口。`/gacha` 显示现有抽卡页面；`/eval` 是新的跨卡池单方案评估页；`/` 保留兼容入口。

`/eval` 不使用左侧导航栏。页面顶部只保留标题和跳转 `/gacha` 的导航按钮，主体内容居中展示。页面先用简短说明介绍评分系统用途，然后进入问卷流程；问卷每次只显示一题，按设计方案先问“倾向”，再问“程度”。高级模式允许直接修改评分参数数值。

## Key Changes

- 页面布局：
  - 新建 `eval.html`，HTML 仍只保留 `#root`，由 JS 写入页面。
  - 顶部栏包含页面标题和“返回抽卡模拟器”导航。
  - 主体使用居中单列布局，电脑端优先，暗色风格与现有页面一致。
  - 页面流程为：系统用途介绍 -> AHP 问卷 -> 高级调参可选 -> 卡池与策略组配置 -> 目标配置 -> 提交评估 -> 结果展示。

- 单方案、多卡池评估：
  - 每次提交一个“整体方案”，整体方案可包含多个卡池阶段。
  - 用户从已有 `configs/config_1` 到 `configs/config_7` 中选择数个卡池，并用可排序列表决定模拟顺序。
  - 每个卡池阶段都有自己的策略组、资源增量、签到开关、源石使用开关等配置。
  - 不做多候选方案横向比较，不调用 `evaluate_multiple_strategies`。

- 策略组表单：
  - 每个卡池阶段使用积木式策略组。
  - 条件块由字段、运算符、比较值组成。
  - 支持字段：`draws/current_up/six_star_count/resource_left/potential/urgent/dossier/soft_pity/up_oprt/oprt`。
  - 每个策略组支持 `全部满足(all)` 或 `任一满足(any)`。
  - 前端将每个阶段策略组转换为 `strategy-protocol-v1`，后端逐个传给 `Scheduler.banner()`。

- 问卷与高级模式：
  - AHP 问卷覆盖四模块：目标达成、期望收益、资源机会、风险稳定。
  - 两两比较共 6 组；每组拆成两步：
    - 第一步：选择更重视哪一项，或同等重要。
    - 第二步：若非同等重要，选择稍微、明显、强烈、极端更重要。
  - 后端或前端按 `1/3/5/7/9` 及倒数计算权重和一致性 CR。
  - CR 超阈值时提示偏好矛盾，但第一版允许继续提交。
  - 高级模式允许直接修改 `ScoringPreferences` 中的权重、角色价值、LogMap 参数、`opportunity_reference`、`risk_utility_weight`、`tail_ratio`、`future_resource_income`、`baseline_samples`、`baseline_seed`。

- API 与任务队列：
  - `GET /api/eval/configs`：返回可选角色卡池配置元数据。
  - `POST /api/eval/jobs`：提交一个整体评估方案，返回 `job_id`。
  - `GET /api/eval/jobs/<job_id>`：查询排队、运行、完成或失败状态。
  - 后端使用进程内任务队列，默认最多 2 个评估任务并行，其余排队。

- 结果展示：
  - 展示一个整体方案的 `raw_score`、评级、四模块分数、目标完成率、平均收益、平均基准、收益倍率、平均资源机会、资源机会倍率、低尾风险、模拟次数、评分版本和关键参数标签。
  - 不展示多策略排名、百分位或与最佳策略差值。

## Test Plan

- 后端：
  - `/gacha`、`/eval` 返回 200，`/` 保持兼容。
  - `GET /api/eval/configs` 能列出已有卡池配置并读取 `char_banner.json`。
  - `POST /api/eval/jobs` 能接收一个整体方案，其中包含多个卡池阶段和各自策略组。
  - 拒绝空卡池、空目标、非法卡池 id、非法条件字段、非法运算符。
  - 小样本任务能完成并返回 `StrategyScoreReport`。
  - 默认并发限制为 2，第三个任务进入排队状态。

- 前端：
  - `/eval` 顶部只有标题和跳转 `/gacha` 的导航，不出现左侧导航栏。
  - 页面先显示系统用途说明，再进入单题问卷。
  - AHP 问卷按“倾向 -> 程度”逐题推进。
  - 高级模式能直接修改数值，并同步到提交 payload。
  - 用户可加入多个卡池、排序、分别配置策略组、配置目标并提交。
  - 页面正确显示排队、运行、成功、失败状态。

- 回归：
  - `uv run pytest test/ -v`
  - `uv run ruff check .`
  - 静态压缩 manifest 能包含新增 JS/CSS，生产模式仍不暴露源码 CSS/JS。

## Assumptions

- “一次只评估一个策略”定义为：一次只提交一个整体跨卡池方案；该方案可包含多个卡池阶段，每个阶段有独立策略组。
- 武器不纳入评分，沿用当前 scheduler 的角色卡池评分范围。
- 第一版策略组 UI 只支持一层 `all/any` 条件组合；复杂嵌套后续再做高级模式。
- 任务队列为单进程内存队列，服务重启会丢失未完成任务。
