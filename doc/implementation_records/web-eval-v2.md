# `/eval` 问卷与策略编辑体验修复计划

## Summary

把 `/eval` 修成一个逻辑清晰的单屏流程页，重点解决四类问题：问卷解释不足、视觉反馈生硬、策略编辑能力不够、页面没有真正铺满窗口。整体方向是把当前“开发者视角的配置页”改成“用户一步一步完成评估”的引导式体验。

本次不改评分后端算法，不改评估 API；主要调整前端流程、文案、样式和策略编辑器的数据建模。策略组嵌套按“两层足够”实现：支持根组 + 一层子组，覆盖 `(A and B) or C`、`(A or B) and C` 这类场景，不做任意深度树编辑。

## Key Changes

### 1. 页面结构与窗口占用
- 页面改成真正的全窗口布局，不再把主要内容限制在中间固定宽度容器里。
- 主体区域使用整窗排版：内容可沿整页宽度展开，仅在具体模块内部控制可读宽度，不保留大面积无用途左右留白。
- 顶部信息区融入页面，不做明显独立 bar；`/gacha` 返回入口保留，但降为辅助层级。
- 页面主体不再被单一外层卡片框死；每一步直接占据页面主要区域，只在局部解释、结果、树节点处使用子面板。
- 首屏大字幅、问卷比较区、结果区都要按“整窗版面”重排，让视觉主轴横向拉开，而不是挤在中央窄列中。

### 2. 顶部信息与文案统一
- 去掉 `ARCHIVE / ORBITAL REVIEW`、`EVAL-01` 等无意义编号文案，保留现有版式气质，但文本统一改为：
  - 主标题：`终末地抽卡策略评估`
  - 副标题：`跨卡池单方案评分器`
- 全页面文案统一改成用户导向，不出现内部实现词汇：
  - `积木式停止策略组` 改成 `停止条件`
  - `全部满足 / 任一满足` 保留，但给出更直白的辅助解释
  - 其他如 `Structured`、`Protocol`、`Dispatch` 之类内部语义从 UI 文案中移除
- 文案按用户理解顺序重写，不再出现“再选程度。”、“映射到 AHP 3”这类面向实现的句子。

### 3. 问卷流程重写
- 问卷首屏先说明系统如何使用问卷，不出现 `AHP` 缩写；改成“系统会通过几道取舍题，判断你更看重什么”。
- 四个模块都补上“是什么 / 影响什么”的说明：
  - `目标达成`：更看重达成你设定的目标，影响目标完成率权重。
  - `期望收益`：更看重平均拿到的内容价值，影响收益分权重。
  - `资源机会`：更看重保留资源给后续卡池，影响资源机会分权重。
  - `风险稳定`：更看重不翻车、低尾更稳，影响风险分权重。
- 每一题改成“情景 + 选择”而不是抽象术语对撞：
  - 先问“这两者必须取舍时，你更不能牺牲哪一项？”
  - 再问强度，但用情景化表达，如“完全不能牺牲”“明显更重要”“只是略微偏向”
  - 内部仍映射到 `1/3/5/7/9`
- `eval-side-label` 的被比较项升级成主标题级视觉，不再像小标签。
- 进度条和题面切换都加过渡动画：
  - 进度宽度平滑变化
  - 题号/状态变化有淡入淡出
  - 进入下一题时题面有轻微切换动画
- 问卷完成后先进入“问卷结果”步骤，不直接跳高级模式：
  - 展示四项权重结果、各项含义、一致性提示
  - 底部给两个动作：`继续评估`、`调整高级设置`

### 4. 高级模式与结果关系
- 高级模式从默认主流程中降级为“可选覆盖层”，不是问卷后的强制下一步。
- 问卷结果页明确说明：高级模式用于手动覆盖问卷生成的数值；不修改则直接沿用问卷结果。
- 高级模式页保留现有参数，但按用户视角重组展示：
  - 权重覆盖
  - 收益价值参数
  - 映射曲线参数
  - 风险与基准参数
- 从高级模式返回时回到“问卷结果/继续后续配置”的分叉点，而不是打断整体流程。

### 5. 输入控件与视觉细节
- 数值输入框去掉浏览器默认上下箭头，统一成暗色自定义输入外观。
- 问卷、结果、策略树节点的动效统一为轻量过渡，不引入额外动画库。
- 当前整窗布局下，要同步重做以下区域的宽度分配：
  - 首屏说明与大字幅
  - 问卷左右对比区
  - 卡池阶段编辑区
  - 目标与提交结果区
- 原则是“整窗占满，但每个局部仍可读”，避免重新退回到窄列表单页。

### 6. 策略组编辑器升级为两层嵌套
- 前端策略编辑器从“平铺条件列表”升级为“两层组编辑”：
  - 根组：`全部满足` / `任一满足`
  - 根组下既可直接挂条件，也可新增一个子组
  - 子组内部可继续添加条件，但不再继续嵌套子组
- 直接支持 `(A and B) or C`、`(A or B) and C` 这类表达。
- UI 用树状/缩进块，不让用户接触 `node_type`、`children` 之类内部结构。
- 前端状态从 `plan.conditions[]` 升级为“根组对象 + 节点数组”；提交时继续序列化为现有 `strategy-protocol-v1` 的 `group/condition` 结构，后端接口不变。
- 旧的平铺条件状态按“根组下全是条件节点”兼容显示。
- 若页面显示策略说明或结果说明，同步保证嵌套组能以中文树状方式清楚展示。

## Public Interfaces / Data Shape

- `/api/eval/configs`、`/api/eval/jobs`、`/api/eval/jobs/<id>` 不改接口路径。
- 评估提交 payload 继续使用现有 `strategy-protocol-v1` structured rule。
- 仅前端本地状态变更：
  - 从“条件数组”改为“根组 + 条件/子组节点”的两层树形编辑状态。
- 若补测试，重点验证：
  - 平铺条件
  - 根组 + 子组
  - `(A and B) or C` 的序列化结果

## Test Plan

- 布局与视觉：
  - `/eval` 在桌面端横向铺满窗口，不再保留大面积无用左右留白
  - 顶部标题区不再像独立 bar，页面主体不再被单个外层卡片限制
  - 首屏、问卷、卡池编辑、结果区都能利用整窗宽度展开
- 问卷行为：
  - 首屏只显示用户能理解的引导，不出现 `AHP` 缩写和开发者式措辞
  - 倾向选择后进入强度选择；每次切换有动画
  - 问卷完成后进入“问卷结果”，而不是直接进入高级模式
  - 从“问卷结果”可直接继续，或进入高级模式再返回
- 视觉与控件：
  - 进度条宽度变化有过渡动画
  - `eval-side-label` 中被比较项在桌面端足够醒目
  - 数值输入框不显示系统默认箭头
- 策略编辑：
  - 可构造根组条件 + 一个子组
  - 可表达 `(A and B) or C`
  - 删除/新增节点后结构仍合法
  - 提交时能正确序列化为 structured group payload
- 回归：
  - 现有评估 API 路由与后端评估测试继续通过
  - 若覆盖策略说明显示，嵌套组输出仍是清晰中文树形文本

## Assumptions

- “占满整个窗口”指主体版面按整窗宽度排布，不再使用中央固定窄容器；局部模块仍可保留必要的可读宽度和内部面板。
- “问卷结束后先看结果”指的是先看“问卷权重结果”，不是立即跑完整策略评估结果。
- 策略组嵌套本次只做到两层：根组 + 一层子组；不做任意深度递归编辑器。
- 高级模式继续保留现有全部参数，只重写文案与分组，不缩减字段集合。
- 动效采用轻量 CSS/前端状态过渡，不引入额外动画库。
